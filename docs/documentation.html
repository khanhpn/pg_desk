Startup Experience

Open PgDesk into a ready workspace

PgDesk starts with a focused splash screen that shows the project logo and loading progress while the Electron workspace initializes behind the scenes.

  • The main workspace stays hidden during initial rendering.
  • The splash remains visible long enough to avoid a distracting flash on fast machines.
  • After the renderer is ready, PgDesk closes the splash and opens the main window maximized.
  • Users do not see a partially rendered workspace behind the loading screen.
Branded Centered PgDesk logo, product name, and status text.
Calm loading Indeterminate progress bar while the app prepares.
Ready reveal Main window appears only after startup is complete.
Connections

Create a PostgreSQL profile

Connection profiles are the entry point for every PgDesk workflow. A profile stores the target server information and lets the app reconnect, refresh metadata, run queries, and perform backup or restore actions against the selected database.

  1. Open PgDesk and create a new connection profile.
  2. Enter host, port, database, username, and password.
  3. Enable SSL when the PostgreSQL server requires it.
  4. Test the connection before saving the profile.
  5. Connect the profile to load schemas, tables, and views.
PgDesk keeps multiple connection profiles and tracks which profile is currently active.
Required fields Host, port, database, user, and password.
Optional security Enable SSL when your PostgreSQL server requires it.
Explorer

Browse schemas and tables

After connecting, the left sidebar loads database metadata in a tree structure. Schemas, table groups, and view groups can be collapsed or expanded so large databases stay readable.

  • Click a table or view to open a quick select query.
  • Right-click a table to open the table action menu.
  • Refresh the explorer when database objects change.
  • Resize the sidebar when names or schema trees need space.

The table context menu is designed for repeated schema work: open table details, inspect columns and indexes, and keep related table actions close to the relation name.

SQL Workspace

Work across saved query tabs

PgDesk supports multiple SQL tabs. New tabs start empty, unsaved tabs show a dirty marker, and saved tabs are restored later with their SQL content. 

select *
from public.users
limit 100;
  • Use the plus button to create a new query tab.
  • Use the close button to remove a tab when more than one tab exists.
  • Use Format to normalize SQL layout.
  • Use Save Tab or the keyboard save shortcut to persist tab SQL.
  • Highlight one or more SQL statements and click Run Selection to execute only the highlighted text.
  • With no highlighted text, Run executes the entire active tab.
Dirty state Unsaved tab edits show a marker beside the tab title.
Persistence Saved tab SQL is restored when the app opens later.
Keyboard save Use Command+S on macOS or Ctrl+S on Windows/Linux.
Run context Selection-first execution keeps unrelated statements from running.
Result Grid

Edit rows from query results

Query results appear in the bottom panel. When PgDesk receives enough table metadata to map a cell back to a table row, the cell becomes editable.

  • Text-like editable values render as inputs.
  • Boolean editable values render as checkboxes.
  • Select multiple rows with the row checkboxes.
  • Use + to append a draft row.
  • Dirty cells and draft values are counted before saving.
  • Save writes all changed values back to PostgreSQL.
  • Delete removes all selected rows after one confirmation.
  • The result panel can be resized vertically.
Editable saves require a base-table result with primary-key metadata. The Add, Save, and Delete controls stay disabled when PgDesk cannot prove that a write is safe.
Editable Cells from known table columns with primary key context.
Read-only Computed expressions, joins without clear identity, or missing metadata.
Save feedback The compact icon toolbar shows saved counts or database errors.
Backup and Restore

Protect database changes

PgDesk creates readable plain SQL backups with pg_dump and restores from selected backup files. Backup filenames use the database name, an incrementing version number, and the current date. 

database_v1_2026_06_30.sql

Backup and restore depend on PostgreSQL client tools. PgDesk searches common locations for pg_dump, psql, and pg_restore, including Homebrew, libpq, Postgres.app, and Windows PostgreSQL folders.

If your tools live somewhere custom, set PGDESK_POSTGRES_BIN to the folder containing pg_dump, psql, and pg_restore. For Docker databases, PgDesk can run the PostgreSQL tools inside the container with docker exec when local tools are not installed.
  1. Select Backup from the active connection card.
  2. Choose the destination SQL file path.
  3. PgDesk runs pg_dump with plain SQL output, either locally or inside the detected Docker container.
  4. Select Restore to choose a SQL backup and run psql. Older custom dump files still restore through pg_restore.
  5. After a successful restore, refresh the explorer metadata.

Help

Troubleshooting

Connection test fails

Confirm the host, port, database, username, password, and SSL mode match the PostgreSQL server configuration. Also confirm the database accepts connections from your machine.

Explorer is empty

Refresh the explorer after connecting and confirm the active database user has permission to read schema metadata.

Result cells are not editable

Editable cells require table metadata, primary key data, and a query result that can be mapped back to a real table column.

pg_dump, psql, or pg_restore is not found

Install PostgreSQL client tools or set PGDESK_POSTGRES_BIN to the folder containing pg_dump, psql, and pg_restore. If the database runs in Docker, make sure Docker is running and the PostgreSQL container publishes the same port used by the PgDesk connection. Set PGDESK_DOCKER_CONTAINER when multiple containers are ambiguous.

Pull request cannot merge

Wait for the PR tests workflow to pass. Repository branch protection can require the Validate check before merge.
